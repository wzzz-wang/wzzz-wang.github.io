Kubernetes官方Java客户端使用操作

发表于 2021-11-29 更新于 2021-12-02 分类于 K8S ， Java

前言

最近终于完成了K8S一体式发布平台的第一个版本的开发，回想一下这其中最复杂也最重要的模块就是K8S的操作模块。这个笔记主要介绍K8S的Java客户端的使用，对于K8S的基础概念就不多介绍了，重点会放在Java代码和API上，至于具体原理可以参考官网或者其他博客。

环境介绍

开发环境

JDk 1.8.0_191
K8S1 1.15
K8S2 1.16
K8S3 1.17
K8S集群版本有三种版本，这里也是这篇笔记重要的地方，我们需要使用同一份代码对三种不同集群进行操作。

需求分析

使用API来写代码其实很简单，但是这其中有很多坑，这里先说一下我们主要使用client-java操作的实体。
我们知道K8S中有三种比较基本的对象，分别是deployment，service，ingress，我们需要使用Java来操作这三种对象，并且实现增删改查的功能。
对于K8S的操作也有很多不同的命令，这里简单介绍一下

以上几种操作命令都是对于K8S中对象的操作，其中apply命令是属于声明式命令，至于具体原理和概念可以自行参考K8S官网，我们这里主要就是实现这上面几种命令的语义。

代码开发

代码开发是非常简单的，只是去调用API就好了，但是调试起来是非常复杂的，这里我就直接贴出来我写的对三种不同对象的操作代码，然后再来细说这其中的坑。

创建K8S_CLIENT

使用client-java来操作K8S集群需要先构建一个ApiClient，这个ApiClient需要通过K8S集群的config文件来生成，这个文件在K8S集群的master节点的.kube文件下，具体位置可以在自己的集群中查找。

deployment操作

以上是deployment的全部操作方法，此处有几个坑

  • 第一坑：序列化和反序列化的问题
    在我部署运行代码的时候想要查看运行结果的时候发现并不能够正常使用Json解析结果，最后才发现K8S官方java客户端是谷歌的，涉及到JSON处理时首选自家的Gson，最终使用GsonBuilder对结果进行序列化，将返回结果进行输出，最终的返回结果也是对应的K8S实体配置，可以更好的观察运行结果。同时返回的结果有很多不需要关注的信息，比如uid、selfLink等等，可以自行筛选删除，上述的K8SCommonUtil.resultFormat方法就是这个作用，这个方法我就不贴出来了，可以自己动手操作实验一下，我已经删除了很多无用信息。
    后面的service操作和ingress操作都有这样的问题就不赘述了。
  • 第二坑：版本问题
    版本问题是一个大问题，不仅仅是Java API的版本问题，还有K8S yaml的版本使用问题，这里分成两块。
    1. 上述代码中create、update、delete和check方法对于K8S集群的1.15版本，1.16版本，1.17版本都可以使用，但是apply方法仅可以在K8S集群1.16版本之后使用
    2. K8S的yaml文件中，deployment的声明文件apiVersion字段仅可以使用apps/v1
      apiVersion: apps/v1，使用其他版本则会报错
  • 第三坑：日志问题
    在实际开发过程中，部署失败方法报错API只会返回一个client-java的方法报错，并不会说明真正的K8S部署报错，这时候需要我们打开client-java的debug模式，这个模式会打印出部署的全部过程以及返回值，具体设置方法就是apiClient.setDebugging(true)，这个debug模式推荐只在开发测试的时候打开，不要在生产环境打开，否则日志会非常的多并且杂乱

上面三个坑对于是属于基础坑，对于deployment，service和ingress都适用，后面就不过多赘述。

service操作

service的操作同样也有很多坑这里来一一说明，关于版本兼容以及方法使用和deployment一样，这里就不过多说明了。

  • 第一坑：在使用create、update方法时有很多需要注意的点，在使用create创建一个新的service的时候，可以不用指定clusterIP和resourceVersion字段，但是在在使用update方法是，一定要与想要更新的service使用当前配置相同的clusterIP和resourceVersion，这两个字段一定要相同，否则会报错，不会正确找到想要更新的service。但是在使用apply方法时，则不用在意这两个字短，这就是声明式发布的好处，无需关心具体的版本号和clusterIP的问题。由于如果要使用update方法去更新service的话，还需要在数据库中维护两个新的字段，我最后选择了先删除在创建的方式，这种方式对于service和ingress来说并不会有影响，但是这种方式不能对deployment使用，这样会有副本丢失以及服务不可用的问题。
  • 第二坑：yaml文件使用的apiVersion一定要是v1，使用别的版本则会报错，无法构建apiVersion: v1

ingress操作

ingress这部分的操作是比较简单的，坑也比较少，主要还是和deployemnt一样的三大坑，这里ingress的yaml使用的是apiVersion: extensions/v1beta1，其他的版本我没有测试过，这个是我们集群统一使用的版本，如果感兴趣可以测试一下其他的版本。ingress的代码涉及的东西比较少，所以我就直接贴了我的生产代码，已经完全封装过了，也没有开启debug模式，其他的两个模块都是测试代码，所以写的不是很规范。

总结和回顾

这一套代码下来已经可以基本覆盖了K8S的基础操作，deployment、service和ingress的增删改查操作，以及对应的aplly操作。下面列出我觉得需要注意和非常重要的地方。

  • 在K8S集群1.15版本，无法使用apply方法，只能使用增删改查操作！！！
  • client-java的序列化使用的是Google的Gson，如果需要查看返回值，则需要使用Gson来进行序列化与反序列化，同时Gson也可以转成Json，也很方便。
  • deployment不能使用删除再创建的操作逻辑，尤其是生产环境，这样会使服务不可用并且丢失副本，无法溯源。
  • 生产环境不要开启debug模式，在本地开发中可以开启，以便查看报错和调试。
  • deployment的yaml文件中使用的版本一定是要apiVersion: apps/v1，否则报错，不会部署成功。
  • service部署如果要使用update方法，一定要使用当前K8S集群中的resourceVersion和clusterIP，如果对应不上则会报错。

以上就是所有的K8S Java Client的操作使用，这些也只是冰山一角，也只是我用到的方法和踩到的坑，这些坑大部分都是因为不同的集群环境不一致导致的，所以在开发公共平台的时候一定要考虑到不同的环境的不同差别，做到尽可能详尽的测试与调试。
