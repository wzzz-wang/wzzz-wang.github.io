SeaTunnel调研

发表于 2022-02-12 更新于 2022-02-13 分类于 SeaTunnel

前言

目前在学习和复习Spark的知识，然后老师说可以调研一下WaterDrop这个组件，然后就去GitHub搜了一下，目前已经改名叫SeaTunnel了，可能他们觉得水滴太小，不太符合他们高性能处理大量数据的调性，改成了SeaTunnel，也许更能表演他们应用的功能和性能。：）在2021年12月9日，SeaTunnel成功进入 Apache 孵化器。值得一提的是这个组件的两个创始人都是中国人。

什么是SeaTunnel

首先在调研这个组件时候要了解什么是SeaTunnel，根据官网的介绍，SeaTunnel 是一个非常易用的支持海量数据实时同步的超高性能分布式数据集成平台，每天可以稳定高效同步数百亿数据。
简单来说，我对SeaTunnel的理解就是一个ETL应用，可以将不通数据源中的数据导入到你想要的存储容器或消息队列中，这其中包括了目前很多主流的存储介质，包括单不限于HDFS，KAFKA，Socket，Druid，ClickHouse等等。
SeaTunnel是依托于Spark构建开发的，Spark在批处理上的性能强大是众所周知的，但是对于Spark的编程是有门槛的，所以SeaTunnel致力于将Spark变得更加简单，更加高效，减少学习成本，加强分布式数据处理的能力。
至于SeaTunnel的数据处理流程，官网上介绍的清晰明了，我这里就不增加大家的阅读障碍了，可以直接移步官网阅读。
接下来我就使用SeaTunnel来做一个简单的数据处理。

环境依赖

作为一个开源的数据集成平台，对环境肯定是有依赖的。
1.Java运行环境 java >= 8
2.Spark集群环境，以下任意一种

  • Spark on Yarn
  • Spark Standalone
  • Spark on Mesos

番外

为什么SeaTunnel不利用Flink来做呢？？？Flink和Spark都是支持流批一体的数据处理框架。事实上SeaTunnel的v2版本是支持Flink的，但是这个v2版本更新停留在了2020年十月13号，就我来看。SeaTunnel的主要目的还是作为一个大批量的离线数据的转储处理平台，能够做一些简单的数据处理工作，类似于将数据打平写入Hive库等操作，对于实时数据处理还是有待发展。

注意⚠️

如果要是使用SeaTunnel读取HDFS文件，需要注意使用的spark依赖要基于读取的Hadoop集群版本构建的，否则读取不到hdfs文件
